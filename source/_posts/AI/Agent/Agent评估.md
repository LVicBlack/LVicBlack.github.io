## Agent评测

这份笔记整理自四类资料：**《AI Engineering》**、**《AI Agents in Action》**、**《Agentic Design Patterns》**，以及美团 LongCat 团队的 **VitaBench** 论文与相关资料。整理完以后，我最深的感受是：**Agent 评估没有一个放之四海而皆准的标准答案**。它必须结合**具体业务场景来设计，业务目标不同、工具链不同、风险边界不同，评测指标和测试方式也会不同**。

**Agent 评估本身也相当复杂**。一个项目如果真的要做一套非常健全的 Agent 评估体系，可能需要一小波人专门投入来设计评测集、指标、自动化 Pipeline、线上监控、红队安全测试和反馈闭环。

这份笔记目的：

1. **先掌握理论框架**：理解 Agent 评估为什么复杂，以及它和传统软件测试有什么不同。

2. **能应对面试问题**：当面试官问到 Agent 评估、测试、上线监控、安全、成本延迟、多 Agent 协作时，你的回答能清晰、有条理、有深度。

3. **能反推项目测试体系**：如果你真的想给自己的项目设计一套比较完善的评估流水线，这份笔记能帮你知道应该从哪些模块入手。

### 1 Agent评估vs传统行业评估

#### 1.1 从TDD到EDD

在传统软件里，测试是写完代码之后的一道工序；但在 Agent 系统里，**评估本身就是开发的起点和方向盘**。

《AI Engineering》提出了一个核心主张——**评估驱动开发（Evaluation-Driven Development）**，它类比于测试驱动开发（TDD）：在你动手搭一个 Agent 之前，应该**先把"怎么算成功"定义清楚**。原因很现实：大模型是概率性的，同样的输入两次运行可能给出不同结果，你无法像传统程序那样"读一遍代码就知道它对不对"，唯一能依靠的就是一套可量化、可重复的评估体系。

这本书甚至直言：**"评估是 AI 工程中最难的问题"**，并用了整整两章（第 3、4 章）来讲它。换句话说，谁掌握了评估，谁才真正掌握了 Agent 的质量。评估的价值贯穿全生命周期：

* **选型时**：靠评估在一堆模型里选出最合适的；

* **开发时**：靠评估判断每次改动是变好还是变坏；

* **上线时**：靠评估做 A/B 测试、灰度放量；

* **生产时**：靠评估（监控）发现线上退化、漂移和安全问题。



***

#### 1.2 Agent评估和传统评估的不同

为什么不能拿传统软件测试那一套来测 Agent？四份资料从不同角度指出了同一件事：**Agent 打破了传统测试赖以成立的前提**。可以从五个维度看清两者的根本差别：

* **确定性**：传统软件测试同样输入必得同样输出、可断言相等；Agent 评测则是概率性输出，同一输入多次结果不同。

* **任务性质**：传统测试是封闭式（close-ended），输出落在预定义值域内（如分类只能输出有限的几个类别）；Agent 评测是开放式（open-ended），同一输入有无数种可能的正确回答。

* **判定方式**：传统测试有唯一/可枚举的标准答案，把输出与它比对、相等即对（可自动断言）；Agent 评测无法枚举所有正确答案，只能用打分、相似度、通过率去度量"好不好"。

* **评测对象**：传统测试看单个函数的返回值；Agent 评测看一条多步"轨迹"：规划→调工具→观察→再规划。

* **失败形态**：传统测试的失败是抛异常、返回错值；Agent 的失败是工具用错、目标跑偏、自以为完成、绕圈子……

最关键的一条是：**传统测试断言"输出 == 期望值"，而 Agent 的输出根本无法这样断言**。《Agentic Design Patterns》明确指出，正因为 Agent 是非确定性的，传统"通过/失败"的测试不足以保证可靠性，必须转向"对效能、效率、合规做持续测量"的思路。

#### 1.3 Agent评估5大困境

综合四份资料，Agent 评测之所以难，集中在五个"坑"上。这五条建议在面试里能直接说出来：

1. **非确定性，传统 pass/fail 失效。** 同一输入多次运行结果不同，无法用"等于期望值"来判定对错。

2. **不能只看结果，要看"会怎么失败"。** 《AI Engineering》提出一个关键视角：**评估 Agent = 识别它的失败模式，并量化每种失败发生的频率**。一个 Agent 可能最终答对了，但中间用错了工具、绕了远路——只看结果会漏掉这些。

3. **失败面更多、更新。** Agent 比单纯的 LLM 多出了规划、工具调用、多轮交互、多智能体协作等环节，每个环节都是新的失败来源。任务越复杂，失败点越多。

4. **Benchmark 会被"污染"，榜单不可尽信。** 公开 benchmark 的题目可能已经混进了模型的训练数据（数据污染 / contamination），导致分数虚高。选型时不能只看排行榜名次。

5. **"裁判"本身也有偏见。** 现在普遍用 LLM 给 Agent 打分（LLM-as-a-judge），但《AI Engineering》系统揭示了裁判模型自身的偏差：**自我偏好**（偏向自己生成的答案）、**首位偏差**（偏向排在前面的选项）、**冗长偏差**（偏向更长的回答）。还有一条警告——**比较式评估里"偏好 ≠ 正确"**：人类投票选出的"更喜欢"，不等于"更正确"，如果投票者不是内行，甚至会把 Agent 训练成"讨好型"而非"正确型"。

### 2 Agent评估全景图

在深入每一种具体的评测方法之前，我们先建立一张“评测全景地图”，让你直观地看清：**一个 Agent 从立项、开发到上线运行，评测到底分布在哪些环节？**

本章是全篇报告的“指引篇”──它先明确解答“哪里要测”**，第 3 章再对仗讲清楚**“怎么测”。

#### 2.1 一条主线

为什么评测会散布在开发的每一个角落？《AI Engineering》在第 1章极富洞见地指出，评测之所以贯穿始终，是因为要达成以下 **四个核心目的**：

这些目的恰好落在开发时间线的不同位置上，把它们半闭环地串联起来，就是一条**完整的评测生命周期**。

这本书还有一个更根本的主张，被称为：

> 💡 **核心观点：评估驱动开发（Evaluation-Driven Development）** 在你动手搭建一个 Agent 之前，**必须先定义“怎么才算成功”**。类比于传统软件的测试驱动开发（TDD），**先定义评估标准，再动手开发；待各维度测法清晰后，再把它们组装成可自动运行的评估 Pipeline。**

因此，评测决不是开发末尾临时拥抱的防黑天鹅工序，而是一路延续到下线的主航道。其余三份资料也各自深度呼应了这条“评估主线”：

* **《Agentic Design Patterns》**：强调评估要“贯穿到线上生产监控，并最终沉淀为可问责 of 工程治理”；

* **《AI Agents in Action》**：直接把“评估”做成了与推理、记忆、规划并列的 **Agent 四大核心组件**之一；

* **美团龙猫**：示范了如何将这套系统性评估思想凝结成一个真实场景下的生活服务评测基准（VitaBench）。

#### 2.2 上线前验证 → 上线后监控

沿着真实交付生命周期，评测不应该简单分成"开发期测质量、上线后测稳定和安全"。更准确的分法是两层：

* **上线前验证**：用离线评测集、仿真环境、压测、红队用例和发布闸门判断"能不能上线"；

* **上线后监控**：用真实流量、日志、trace、metrics、用户反馈判断"上线以后有没有退化、漂移、成本失控或被攻击"。

成本、效率、延迟、安全、合规不是生产期才出现的指标，而是**横向贯穿两层**：上线前它们是准入门槛，上线后它们是持续监控项。

**🛠️ 贯穿全程的「通用武器库」（对应 §1）**

在探讨具体阶段前，必须先掌握三件"通用武器"（LLM-as-a-judge、比较式评估、基础度量）。它们没有固定归属，而是在模型选型、组件调试、发布闸门和生产监控里被反复调用。

**✅ 上线前验证：回答"能不能上线"（对应 §2 \~ §6）**

上线前不是只测"答案对不对"，而是要把模型、组件、整机、资源消耗和安全红线一起放进发布闸门。

* **选型期：挑对地基模型（对应 §2）**
  公开榜单只作粗筛；用 License、限价、延迟、隐私等硬约束缩小候选池；最后用业务私有数据集终筛。这里已经要看成本和延迟，不能等上线后才发现用不起、太慢。

* **开发期前置标准：先定义"怎么算好"（对应 §3）**
  按《AI Engineering》的四类标准圈定方向：领域能力、生成能力、指令遵循、成本与延迟；同时补上 Agent 项目必须有的安全/合规红线。

* **组件评测：逐个测零件（对应 §4）**
  Prompt、RAG、工具调用、规划推理分别隔离评估。Prompt 安全、RAG 忠实度、工具乱调、规划绕路，都属于上线前必须暴露的问题。

* **系统评测：测整机能不能稳定完成任务（对应 §5）**
  用端到端成功率、轨迹评测、多轮用户模拟、多 Agent 协作评测，检查系统整体可靠性。效率也在这里进入评分：步数过多、绕远路、超时，不能算真正可用。

* **评测集 + Pipeline + 发布闸门（对应 §6）**
  把前面各维度的测法组装成可自动回归的评估流水线，并设上线阈值：质量分、Pass^k、P95 延迟、token/成本预算、最大步数、安全红队通过率、误拒率等。

**📡 上线后监控：回答"真实流量下有没有变坏"（对应 §7）**

上线不是开始测试成本和安全，而是把上线前已经定义好的指标迁移到真实流量里持续观测。

> 🎯 **警言：监控的目标和评估的目标，本质上是同一个。** 区别在于：上线前主要用准备好的数据、仿真和压测；上线后用真实用户流量、日志、trace、metrics 和反馈。

* **A/B 测试与灰度发布**：小流量分发，用留存率、转化、满意度、投诉率等线上业务指标作终审，并设置回滚线；

* **监控与可观测性**：用 logs、traces、metrics 记录每次运行，并在这套观测底座上持续监控质量、漂移、异常、成本、延迟、安全和合规。

把两层主线连起来读，就是一个 Agent 评测的完整故事：

> 💡 **全景流串联：** **上线前先把"能不能上线"测透**：选模型 → 定成功标准 → 测组件 → 测整机 → 组装评测集和 Pipeline → 用质量、成本、延迟、效率、安全、合规做发布闸门；
> **上线后再把同一组指标搬到真实流量里持续监控**：A/B 灰度 → logs/traces/metrics → 漂移和异常检测 → 成本/延迟/安全告警 → 回灌评测集重新回归。



越往后走，评测的场景便越具体，直接接触生产中极易在 Edge Cases 上坍塌的“深水区”。**接下来的章节，就把这条主线上的每个环节逐一摊开，讲清每一处"怎么测、用什么工具"。**

### 3 评测细节

本章是全篇主体。我们沿着第 2 章那张生命周期主线，**从"通用武器库"开始，再依次走过上线前验证（选型 → 前置定成功标准 → 组件 → 集成 → 组装评测集与 Pipeline → 发布闸门）和上线后监控（灰度、可观测性、漂移、成本与安全告警）**。每一个小节都用同一个模板讲：

> **【是什么】** 这一节评的对象是什么 → **【为什么测】** 不测会怎么翻车 → **【怎么测】** 具体指标、工具、可照做的步骤。
>
> 📌 这里的怎么测，注意也不是标准的答案。而是我基于这4个资料来总结的。怎么测这一步有的地方写的是具体具体的思路，有的地方是一些可以参考的做法。具体真正工程实践上，没有标准答案，而是要根据我们今天讲解的思路，结合自己的项目灵活扩展。

#### 1 通用武器库

在进入各个生命周期阶段之前，先认识三件"通用武器"。它们不属于某一个阶段，而是**在选型、组件、集成、生产里被反复使用**——先讲清楚，后面各节直接引用。

##### 1.1 LLM-as-a-judge

**【是什么】** 用一个（通常更强的）大模型，按照预先写好的评分标准（rubric），去给另一个 Agent 的输出打分或排序。它是目前**最主流的自动评估手段**，四份资料无一例外都在用。

它有三种典型用法：

* **单点打分**：给一个输出打一个绝对分（如"有用性 1–5 分"）；

* **对参考打分**：把输出和"标准答案"比，判断是否一致；

* **两两比较**：给出 A、B 两个输出，让裁判选哪个更好。

**【为什么测】** Agent 的输出大多是开放式自然语言，没有唯一标准答案，用精确匹配根本没法判分；纯人工评估又太慢太贵、无法规模化。LLM-as-a-judge 填补了中间地带——它能像人一样理解语义，又能像程序一样批量自动跑。《AI Engineering》给了一个有说服力的数据：**GPT-4 作为裁判与人类的一致性可达 85%，甚至高于人类评审之间的一致性（81%）**。

**【怎么测】**

> 定性：这是**一个方法、分 3 步走**（写标准 → 固定裁判 → 批量跑），三步是**先后顺序**，不能跳。

1. **【步骤】写清楚 rubric**：定义评估维度和每个分档的含义。实战书的做法是借用教育学的"评分量规"概念，比如评一个推荐器，从 `subject / format / genre` 三个维度各打 1–5 分，并写明每一分代表什么。

2. **【步骤】固定裁判模型和提示词**：设计模式书里用 Gemini（`gemini-1.5-flash-latest`，`temperature=0.2`）搭了一个 `LLMJudgeForLegalSurvey`，并用 `response_mime_type="application/json"` 强制输出 JSON 分数（含 `overall_score`、`rationale`、`detailed_feedback`、`concerns`、`recommended_action` 五个字段），方便后续解析聚合。

3. **【步骤】批量跑 + 聚合**：把多组输入喂进去，解析每条打分，再求平均/分布。

**📋 一个真实的 judge 提示词长什么样（直接照搬一个例子）**

下面是实战书第 9 章"推荐器"案例里**真实的裁判提示词**（`evaluate_recommendation.jinja2`，Jinja2 模板，`{{ }}` 里是运行时会被填入的变量）。它麻雀虽小五脏俱全，正好凑齐一个标准 judge prompt 的四要素——**角色设定 + 评分维度 + 1–5 分量规 + 指定输出格式**：

> （`{{subject}}/{{format}}/{{genre}}` 是评测维度，`{{custom}}` 是留给"特殊评分说明"的插槽，`{{recommendation}}` 填入待评的推荐内容。）

**从这个例子能学到三件事：**

* **维度可参数化**：把评测维度写成 `{{ }}` 变量，换个场景填不同值就能复用同一套裁判，不必每次重写；

* **量规要逐档下定义**：不是光说"1–5 分"，而是把"1 分长什么样、5 分长什么样"逐档写清——这正是 AI Engineering 强调的 "create scoring rubrics with **examples**"；

* **要给输出格式做示范**：提示词末尾用 `Title/Subject/Format/Genre` 明确示范了期望的输出长相，裁判才会稳定吐出可解析的结果，方便后续批量聚合。

> 💡 **再补一个佐证"裁判没有统一标准"的发现**：AI Engineering 指出，同样叫"忠实度（faithfulness）"这一个指标，三个主流评估工具的 judge 提示词标度各不相同——**MLflow 用 1–5 分、Ragas 用 0/1、LlamaIndex 让模型直接答 YES/NO**。这意味着 LLM-judge 的提示词至今没有行业标准，**换一个工具或提示词，分数就不可比**。

**⚠️ 必须知道的三大偏见（面试高频）** —— 《AI Engineering》系统揭示了裁判模型自身的偏差，用它的时候要警惕：

* **自我偏好（self-bias）**：裁判偏向自己生成的答案；

* **首位偏差（first-position bias）**：两两比较时偏向排在前面的那个；

* **冗长偏差（verbosity bias）**：偏向更长的回答，哪怕没更好。

> 书里有一句金句值得背：**"看不到裁判模型和它的提示词，就不要相信它给的分数。"** 这意味着任何引用 LLM-judge 结果的榜单，如果不公开裁判细节，可信度都要打问号。

##### 1.2 比较式评估

**【是什么】** 不给单个输出打绝对分，而是让多个模型/Agent 的输出**两两对决**，根据胜负用类似国际象棋的 **Elo 评分**算出排名。最著名的就是 Chatbot Arena。

**【为什么测】** 有些质量很难打绝对分（"这个回答到底算 3 分还是 4 分？"很主观），但人类**比较**两个谁更好却相对容易、也更稳定。比较式评估利用了这一点，更适合给一批模型排座次。

**【怎么测】**

> 定性：**一个方法、分 2 步**——先攒"两两对决"的胜负数据，再用算法换算成排名分。

* **【步骤】收集对决数据**：收集大量"同一问题、不同模型的两个回答"，让人或 LLM 投票选更好的；

* **【步骤】换算排名**：用 Elo / Bradley-Terry 类算法把胜负记录换算成排名分。

**⚠️ 两个关键提醒：**

* **要和 A/B 测试区分开**：A/B 测试是把不同版本分给不同用户群看线上指标；比较式评估是**同时拿多个输出给同一个评委比**，两者不是一回事。

* **"偏好 ≠ 正确"**：投票选出的"更受欢迎"不等于"更正确"。如果投票人不是该领域内行，可能把 Agent 带偏成"讨好型"——回答让人舒服但其实是错的。所以**评委的专业度至关重要**。

##### 1.3 基础度量：精确评估与相似度

**【是什么】** 一组**不依赖大模型裁判**的、更传统也更廉价的度量手段，适合有明确答案或可程序化判定的场景。

主要分三类：

* **功能正确性（Functional Correctness）**：直接看任务有没有真的完成。比如代码题就跑测试用例（HumanEval、MBPP），text-to-SQL 就执行 SQL 看结果对不对——**这是最硬、最可信的指标**。

* **与参考答案的相似度**：

  * *词汇级*：精确匹配、BLEU、ROUGE（看字面重叠）；

  * *语义级*：BERTScore、嵌入向量余弦相似度（看意思接近）。设计模式书还补充了 Levenshtein 编辑距离、Jaccard 相似度等。

* **语言建模指标**：交叉熵 / 困惑度（perplexity）。除了衡量语言流畅度，《AI Engineering》还点出一个巧妙用途——**用它检测数据污染**（如果模型对某测试集 perplexity 异常低，说明可能见过这些题）。

**【为什么测】** 不是所有场景都值得动用昂贵的 LLM-judge。**只要任务有可程序化验证的标准答案（代码、SQL、数学、分类），就应该优先用功能正确性**——它客观、便宜、可重复，是最可信的一类证据。

**【怎么测】**

> 定性：这里不是步骤，而是**按"答案有多可验证"三选一档**——越靠上越客观越该优先用。

* **【档位·最严】有标准答案**（分类、抽取、SQL、代码）→ 用功能正确性 / 精确匹配；

* **【档位·中】答案灵活但有参考**（翻译、摘要）→ 用 BLEU/ROUGE + BERTScore；

* **【档位·最松】完全开放**（对话、创作）→ 退回到 1.1 的 LLM-as-a-judge。

> **选型口诀**：**能用功能正确性就别用相似度，能用相似度就别用 LLM-judge**——**越靠前越客观越便宜**。三者常常组合使用，因为没有任何单一指标能覆盖 Agent 的全部能力。

#### 2 选型期：选基模

**【是什么】** 在动手开发之前，从一堆候选大模型里评测、挑出最适合你这个 Agent 任务的那个（或那几个）。

**【为什么测】** 模型是 Agent 的"地基"，选错了后面再多优化也补不回来。但选型最大的坑在于——**不能只看排行榜名次**。《AI Engineering》专门用一节讲选型的陷阱：公开榜单可能存在**数据污染**（测试题混进了训练集，导致分数虚高）、榜单任务和你的真实业务未必相关、闭源模型还涉及成本与数据隐私的权衡。

**【怎么测】**（一套可照做的选型工作流）

> 定性：**一个方法、一条"五步漏斗"**——从粗到细一层层筛。其中第 4 点是漏斗每一层都要看的**考查维度**（不是独立一步），夹在步骤里说明。

1. **【步骤】区分硬性约束和软性偏好**：先用硬指标（成本上限、延迟上限、是否支持私有部署、上下文长度）筛掉不合格的；剩下的再比软指标（质量、能力）。

2. **【步骤】公开榜单只作初筛**，并对它保持警惕：查这个 benchmark 有没有被污染、任务是否贴近你的场景。

3. **【步骤】建私有评测集做终筛**：用你自己业务的真实数据（公开榜单看不到的）跑一遍，这才是最可信的。

4. **【维度】每一层都要覆盖的考查点**：正确性、成本、延迟，以及领域专项能力。

5. **【步骤】build vs buy 决策**：自建开源模型 vs 直接调 API，按成本、隐私、可控性权衡。

#### 3 上线前：组件评测

Agent 是由多个组件拼起来的：提示词、检索、工具、规划。这一阶段**把每个零件单独拎出来测**，因为只有先保证零件合格，整机才可能合格。

##### 1 Prompt 评测

**【是什么】** 评估提示词（prompt）写得好不好、以及它扛不扛攻击。

**【为什么测】** 提示词是 Agent 行为最敏感的"控制旋钮"，改一个词输出可能天差地别。但 prompt 的好坏不能靠"感觉读着顺"，得有系统化的对比；同时 prompt 还是安全攻击（越狱、注入）的主要入口，必须做防御测试。

**【怎么测】**

> 定性：**3 个相互独立的方法**，分别测 prompt 的"好不好用"、"能不能自动调优"、"扛不扛攻击"——按需要取用。

* **▶ 方法一 · 质量评测：用微软 Prompt Flow 做批量对比**。这是实战书第 9 章的主角，本身是一条**分 4 步**的落地流程：

  1. **【步骤】** 在 VS Code 里用 Prompt Flow（DAG 可视化流程图）搭一条评估流；

  2. **【步骤】** 用 JSONL 准备多组输入，批量跑不同的 prompt 变体（比如把参数注入到 system prompt 还是 user prompt）；

  3. **【步骤】** 用 LLM-as-judge 按 rubric 给每个变体的输出打分；

  4. **【步骤】** 用 `aggregate.py` 聚合各 run 的平均分、`log_metric` 记录指标，可视化对比哪个 prompt 更好。

* **▶ 方法二 · 自动 prompt 优化工具评测**：DSPy、TextGrad 等能自动搜索最优 prompt，但要警惕它们隐藏的 API 调用成本和模板 bug。

* **▶ 方法三 · 防御性评测**：主动测 prompt 的鲁棒性（轻微扰动看输出是否崩）和安全性（上线前进 发布闸门，上线后进 持续监控）。

> 落地抓手一句话：**Prompt Flow + JSONL 批处理 + LLM 打分 + 聚合对比**，把"调 prompt"从玄学变成可量化实验。

##### 2 RAG / 检索评测

在笔记 [RAG 评估章节](https://my.feishu.cn/docx/GikId1snboM6LyxxGPXc0kDdnub#doxcnJnH8EjFWiQscDRABrYV3Ac)，有更详细的专门讲RAG的评估内容。

**【是什么】** 评估 Agent 的"外挂记忆"——检索增强生成（RAG）——找回来的资料质量好不好。

**【为什么测】** RAG 是"先检索、再生成"。如果检索这一步找回来的就是错的/不相关的，那后面生成得再流畅也是"一本正经地胡说八道"。所以 RAG 的评估要**拆成检索和生成两段分别看**。

**【怎么测】**（一组成熟的检索指标）

> 定性：不是并列的几个方法，而是**把 RAG 切成"检索段 + 生成段"两段**，每段列出要看的**考查维度**。前四个是检索段维度，第五个是生成段维度，最后一个是实战书的动手实现。

* **【维度·检索段】context precision（上下文精确率）**：找回来的内容里，有多少是真正相关的；

* **【维度·检索段】context recall（上下文召回率）**：所有该找回的相关内容，找回了多少；

* **【维度·检索段】排序质量**：用 NDCG、MAP、MRR 衡量相关内容是否排在前面；

* **【维度·检索段】嵌入模型质量**：用 MTEB、BEIR 这类基准评测你用的 embedding 模型好不好；

* **【维度·生成段】生成段的忠实度（faithfulness）**：生成的回答是否真的"扎根"于检索到的内容、有没有脱离资料瞎编。

* **【实现】工程实操**：实战书第 8 章演示了直接看**向量相似度分数**、调 **top-n** 数量、试不同 **chunk 切分策略** 对检索准确率的影响——这些都是能立刻上手调的旋钮。

##### 3 工具调用评测

**【是什么】** 评估 Agent 调用外部工具（API、函数、数据库）的能力——选对工具没有、参数填对没有、调用顺序对不对。

**【为什么测】** 工具调用是 Agent 区别于普通聊天机器人的核心能力，也是高频失败点。注意它有**三个不同层次**，面试时能区分开会加分：

* **能力层**：在很多工具里能不能选对、调对（龙猫：66 个工具 + 512 条依赖边的工具图）；

* **行为层**：调用工具的顺序对不对（属于轨迹评测，见 5.2）；

* **代码层**：工具函数本身写得对不对（实战书第 5 章用单元/集成测试逐个验证 semantic function）。

**【怎么测】**

> 定性：**一个方法、分 3 步**（造考题 → 跑 Agent 逐项核对 → 补反例），三步按顺序走；其中第二步的"判分"内部有**两个档位**可选。

核心思路是：**造一批"考题"，让 Agent 去调工具，再逐项核对它调得对不对。** 这套方法可以参考业界权威的 Berkeley Function-Calling Leaderboard（BFCL，一套约 2000 道题的工具调用评测基准）——但要注意，BFCL 是**选型期**（§2）用来给各家**模型**的通用工具调用能力排名的，**测的不是你自己的业务工具**。我们要做的，是**借用它的判分方法，把题目换成自己项目的工具**。

把这样的用例攒成一个小数据集（几十条，覆盖常见、易错、和"不该调工具"的情况）。

把①②喂给 Agent，它会输出一个工具调用；判分器拿它和③标准答案**像填检查表一样逐项核对**：① 工具名选对没 → ② 必填参数齐不齐 → ③ 有没有瞎编不存在的参数 → ④ 参数值对不对。任何一项错，这条就算失败，而且能精确定位错在哪一项。

* **看真跑出来的结果对不对**：真的执行 `check_weather`，检查它的返回值。能抓住"调用写得对、但结果其实是错的"这类问题，最贴近真实，但会产生真实副作用、需联网、且结果可能随环境波动。

* 实战中常**两者搭配**：多数用例只核对调用本身，关键工具再补几条真执行兜底。

##### 4 规划 / 推理评测

**【是什么】** 评估 Agent 的"大脑"——它能不能把一个复杂任务拆解成合理步骤、并在执行中正确推理。

**【为什么测】** 先划清和上一节的界线：**&#x20;工具调用看的是"某一次调用填得对不对"（单点动作）， 规划/推理看的是"整条决策链对不对"（全局思路）。** 一个管"手"，一个管"脑"——Agent 可能每一步调用都很标准，却从一开始就想错了该做什么。

《AI Engineering》把规划失败拆成三类，正好告诉我们"要盯哪些错" ：

* **该不该用这个工具 / 这一步**：在决策层面选错了工具或多走了无谓的一步（注意：选哪个工具属于这里；至于参数填没填对，归 4.3）；

* **目标失败**：没达成目标，或达成了但**违反了约束**（书里特别强调"时间"约束最容易被忽略，比如约到了营业时间之外）；

* **反思错误**：任务没真完成，Agent 却**误以为完成了**就停手。

**【怎么测】**

> 定性：**一个方法、分 3 步**（造带"理想步骤链"的考题 → 跑 Agent 对照三类失败核对 → 统计并借力三本书的机制）。第三步底下三个点是**三份资料各自的实现**，并列、可任选增强。

核心思路是：**给 Agent 一个需要分几步才能完成的任务，记录它实际走的"步骤链"，再和"理想步骤链"对比，看它有没有想错。** 和 4.3 不同——这里不抠单次调用的参数，而是看整条路对不对。

**【步骤】第一步：制作测试集——每条用例长这样**

以智能预约 Agent 的"约按摩"任务为例，一条用例记录的是**理想步骤序列 + 约束**：

**【步骤】第二步：跑 Agent，对照三类失败逐一检查**

让 Agent 真的去执行，记录它实际走了哪几步，然后对照三类失败核对：

* **步骤链对吗**：看实际步骤 vs 理想步骤。典型失败——没"查档期"就直接"确认下单"，属于步骤缺失。

* **守住约束吗**：看有没有违反约束。典型失败——约到了晚上 20:00（超营业时间），属于违反约束。

* **反思对吗**：看是否真完成才收尾。典型失败——技师没匹配上，却回复"已约好"，属于自以为完成。

**【步骤】第三步：统计 + 借力**（下面三点是三份资料各自的实现，可任选叠加）

* **【实现·AI Eng】失败模式频率统计**：评估本质是"识别失败模式 + 数每种出现多少次"。常用指标：有效计划占比、平均要试几次才产出有效计划。

* **【实现·龙猫】复杂度可控的 benchmark**：龙猫用"推理复杂度"维度（观测空间大小、部分可观测程度、推理点数量）量化任务有多难，从而系统地考规划。

* **【实现·实战书】推理自评机制**：实战书第 10 章用 self-consistency（生成多条推理路径、投票选最一致的）和 tree-of-thought 来提升并间接评估推理质量。

第二步只标出了"这条用例哪里错了"，第三步要把它**变成可比较的数字**。分三件事，前两件是"怎么打分"，最后一件是"怎么把题出难"——别混在一起：

**(1) 给单条路径打分——两种判法，按有没有标准答案选**

* **和标准答案比相似度**（实战书第 10 章的实际做法）：把 Agent 最终答案和 `expected` 都转成向量，算语义相似度，得一个 0\~1 的分（书里样例就直接输出 `evaluation_score: 0.957`）。适合有明确标准答案的题。

* **看多条路径自不自洽**（self-consistency，**不需要标准答案**）：对同一任务让 Agent 生成 N 条推理路径，然后

  * **多数投票选答案**：N 条结论里哪个出现得最多就采纳它，离群的那条丢掉（书里例子：3 条里 2 条一致，就取那 2 条）；

  * **一致性 = 可信度**：多数有多集中本身就是分数——大多数路径都指向同一答案 → 推理稳、可信；各说各话没有多数 → 说明没想清、可信度低。

**(2) 汇总成整体指标**：评估本质是"识别失败模式 + 数每种出现多少次"。把很多条用例的结果汇总，算：有效计划占比、平均要试几次才产出一个有效计划、各类失败（步骤缺失/违反约束/自以为完成）各占多少。

**(3) 让题目更难更系统**（属于"出题"不是"打分"）：用"推理复杂度"维度（观测空间大小、部分可观测程度、推理点数量）拧难度旋钮，系统地造出从易到难的题，避免"题太水、高分也说明不了能力"。

> 一句话记忆：规划评测 = **对照"理想步骤链"，盯三类失败（步骤选错 / 违反约束 / 自以为完成）→ 给路径打分（和标准答案比相似度，或用 self-consistency 多数投票+一致性）→ 汇总成频率指标**。龙猫的复杂度维度负责把题出难。它看的是整条路，4.3 看的是路上的每一步调用。

***

#### 4 上线前 · 集成 / 系统评测

零件合格不等于整机合格。这一阶段把 Agent 当成一个整体，看它端到端能不能真的把任务办成。

##### 4.1 任务结果评测

**【是什么】** 最直接的一问：给 Agent 一个完整任务，它最终**做成了没有**。

**【为什么测】** 这是用户唯一真正关心的东西。但"做成没有"的判定方式，三份资料因**目的不同而选了不同严格度**，这点很值得讲（面试亮点）：

* **龙猫**：用**全有或全无**——Pass@k / Pass^k（跑 k 次有一次成功 / k 次全成功）。因为要给模型严格**排名**，必须能拉开差距。

* **实战书**：用**1–5 分级**（rubric 打分）。因为开发期要**诊断**哪里弱，需要颗粒度。

* **设计模式**：用**语义相似度 / 忠实度**（连续分）。因为要**持续监控**漂移，需要连续信号。

> 关键认知：**这不是矛盾，而是"目的决定方法"**——排名要严、诊断要细、监控要连续。用错了场景（比如拿 Pass^k 做日常开发诊断）就会觉得"怎么全是 0 分"。

**【怎么测】**

> 定性：判分方法不是并列的方法，而是**按"你的目的"选一档严格度**——目的决定方法（接上表）。

* **【档位·要排座次/狠考】**：用 Pass@k / Pass^k，温度固定，每个任务重复多次。龙猫用 **Pass^4**（同一任务跑 4 次必须全对）来逼出模型的稳定性。

* **【档位·要诊断改进】**：用 rubric 多维度 1–5 分，看具体哪一维拉胯。

***

##### 4.2 轨迹评测

**是什么】** 不只看终点，还要看 Agent **走到终点的整条路径**——它依次做了哪些动作、调了哪些工具、顺序对不对。这是设计模式书讲得最系统的部分。

**【为什么测】** 一个 Agent 可能"蒙对了"答案，但过程一塌糊涂（多绕了 10 步、调了不该调的工具）；也可能答案略有瑕疵但过程完全正确。**只看结果会误判**，轨迹评测能看出 Agent 是不是"真的会"，也能定位它在哪一步出错。

**【怎么测】**

> 定性：这里其实是**两个独立方法**。方法一"轨迹比对"内部有 **5 个档位**（从最严到最松，按需求选）；方法二是实战书的"Judge-Verifier 回路"，另起一种思路。

* **▶ 方法一 · 把实际轨迹和标准轨迹做比对**，松紧分 5 档：

  * **【档位·最严】精确匹配**：动作序列和标准轨迹完全一致；

  * **【档位】按序匹配（in-order）**：关键动作按正确顺序出现即可，允许中间有别的；

  * **【档位】任意顺序匹配（any-order）**：关键动作都出现就行，不计顺序；

  * **【档位】精确率 / 召回率**：实际动作里有多少是该做的、该做的做了多少；

  * **【档位·最细】单一工具使用**：单独看某个关键工具用得对不对。

* **▶ 方法二 · Judge-Verifier 回路**：实战书第 6 章用行为树搭了"Hacker→Judge→Verifier"三段式——Judge Agent 评判解答，Verifier Agent 在独立线程**真的把代码跑一遍**做二次验证。这是轨迹+结果双重把关的好例子。

##### 4.3 多轮交互评测

**【是什么】** 评估 Agent 在**多轮对话**中的表现——用户会追问、改主意、说得含糊，Agent 能不能跟得住。这是龙猫几乎**独家**深入的领域。

**【为什么测】** 真实用户不会一次把需求说全。前面的组件评测、单轮评测都假设"一问一答"，但实际场景是"边聊边改"。如果不专门测多轮，上线后才会发现 Agent 一到真实对话就乱套。

**【怎么测】**

> 定性：这是一套完整流程，不是三种方法任选。核心是：**固定"环境 + 隐藏用户目标 + 评分 Rubric"，让用户模拟器动态生成多轮对话，被测 Agent 正常应对，最后用终态 + Rubric 判断任务是否完成。**

**第一步：制作多轮评测数据集，但不是写死对话文本**

多轮评测的样本不是"用户问一句、标准答案一句"，而是一个任务包。它通常包含三件固定内容：

* **环境 / 后台状态**：被测 Agent、用户模拟器、判分器都可用，作用是提供菜单、库存、价格、时间槽等客观世界。

* **隐藏用户目标卡**：只给用户模拟器，作用是规定用户真正想完成什么、何时改主意。

* **评分 Rubric**：只给判分器，作用是规定最后怎么判 Pass / Fail。

被测 Agent **看不到隐藏用户目标卡，也看不到评分 Rubric**。它只能像真实线上一样，根据用户模拟器一句一句说出来的话来追问、推荐、调用工具、确认和下单。

**第二步：运行时由用户模拟器动态生成多轮对话**

**第三步：对话结束后，用"终态 + Rubric"判分**

判分不是拿整段对话去比一段"标准对话"，而是看最后状态和关键行为是否满足 Rubric：

最后可以像龙猫那样用 **Pass^4**：同一道任务跑 4 次，每次对话都可能不同，4 次都通过才说明 Agent 真稳定。

> 一句话记忆：多轮交互评测不是固定"标准对话"，而是固定**环境、隐藏目标和评分标准**；随机的是用户模拟器生成的对话路径；最终判的是**任务终态和关键行为**是否满足 Rubric。

##### 4.4 多 Agent 协作评测

**【是什么】** 当系统由多个 Agent 组队完成任务时，评估这个"团队"协作得好不好。

**【为什么测】** 多 Agent 系统多出一层全新的失败面——它们之间会不会有效协作、有没有选对该干活的 Agent、加 Agent 到底是帮忙还是添乱。单 Agent 的指标测不出这些"团队动态"。

**【怎么测】**

> 定性：**"协作好不好"不要抽象地打一个总分，而要拆成三套可执行测法**：①对照标准流程看交接对不对；②做消融实验看每个 Agent 有没有真实贡献；③用质检 Agent / trace 抓过程病征。三套可以叠加使用。

先给一个具体例子。假设智能预约系统有 4 个 Agent：

一条测试任务可以写成这样：

**方法一：交接轨迹评测，看"团队配合有没有走对"**

**怎么实现：**

1. **准备评测集**：每条 case 不只写最终答案，还要写 `expected_handoffs`，也就是谁应该把什么信息交给谁。

2. **跑完整多 Agent 系统**：记录每次 Agent 调用、输入、输出、交接对象，形成 trace。

3. **对照标准交接链判分**：逐项检查有没有选对 Agent、有没有传对关键字段、顺序是否合理、最终状态是否正确。

**看哪些指标：**

* **Agent 选择正确率**：该查天气时有没有交给 Weather Agent，而不是通用问答 Agent。

* **交接字段完整率**：日期、时间、服务项目、用户约束有没有传丢。

* **交接顺序正确率**：是否先查天气再改日期，是否在查档期前就贸然下单。

* **端到端成功率**：最终预约状态是否满足任务目标。

**能定位什么问题：** 如果最终约错了日期，trace 能告诉你到底是 Router 没调用 Weather Agent，还是 Weather Agent 查对了但 Router 没把"下雨→改后天"传给 Calendar Agent。

**方法二：消融评测，看"多加一个 Agent 到底有没有用"**

**怎么实现：** 固定同一批 evalset，跑几组对照实验：

* **Full Team**：Router + Weather + Calendar + Booking 全部启用，用来看多 Agent 的完整表现。

* **No Weather**：去掉 Weather Agent，天气逻辑由 Router 或主 Agent 自己处理，用来看天气 Agent 是否真的贡献准确性。

* **No Router**：去掉 Router，所有任务直接给主 Agent，用来看路由 Agent 是否减少误分发。

* **Single Agent**：一个大 Agent 全包，和多 Agent 架构做基线对比。

**看哪些指标：**

* **成功率差值**：`Full Team 成功率 - No Weather 成功率`，差值大说明 Weather Agent 有价值。

* **成本 / 延迟差值**：多 Agent 成功率提高了，但是否多花太多 token、调用太慢。

* **错误类型变化**：去掉某个 Agent 后，是工具错、计划错、交接错，还是最终状态错。

* **边际收益**：增加第 N 个 Agent 后，成功率是否继续升，还是开始下降。

**能定位什么问题：** 如果 Full Team 比 Single Agent 成功率只高 1%，但 token 成本翻倍，说明多 Agent 架构可能过度设计；如果 No Router 版本误调工具率大幅上升，说明 Router Agent 是必要组件。

**方法三：过程监控 + 质检 Agent，看"协作过程中有没有病征"**

**怎么实现：**

1. **给每次多 Agent 协作开 trace**：用 AgentOps 或类似 tracing 工具记录每个 Agent 的输入、输出、耗时、token、调用次数。

2. **配置质检 Agent / Reviewer Agent**：让它在每一步或每几步后检查当前状态，例如"是否已经满足用户约束？是否有 Agent 在重复搜索？是否应升级人工？"。

3. **把质检结果结构化输出**：例如 `handoff_ok / repeated_reasoning / missing_context / wrong_agent / should_stop`。

**看哪些指标：**

* **重复思考 / 重复调用次数**：同一个 Agent 反复查同一件事，说明团队在打转。

* **无效交接次数**：下游 Agent 收到的信息不够，必须追问上游。

* **冲突次数**：两个 Agent 对日期、价格、状态给出不一致结论。

* **人工兜底触发率**：质检 Agent 判定继续自动执行风险太高。

* **每个 Agent 的 token / latency 占比**：找出最贵、最慢、最容易拖垮团队的角色。

**能定位什么问题：** 如果 trace 显示 Calendar Agent 连续 5 次收到缺少 `duration` 的请求，问题不在 Calendar，而在上游 Router 没抽取完整槽位；如果 Reviewer 多次打出 `should_stop=false`，说明 Booking Agent 过早下单。

> 一句话记忆：多 Agent 评测不是"问裁判这个团队好不好"，而是 **测交接链是否正确、做消融看每个 Agent 是否有贡献、看 trace/质检 Agent 抓协作病征**。最终报告里要同时给出端到端成功率、交接正确率、Agent 选择正确率、成本延迟和主要失败类型。

#### 5 上线前收口:Pipeline 与发布闸门

走到这里，上两章 已经把"每个组件怎么测、整机怎么测"都摸清了。现在才是把它们**收口成一条可自动运行的评估流水线**的时候——这一步同时包含两件本就该一起做的事：**建评测集**和**搭 Pipeline**。把它们合成一步，是因为评测集里每条样本的"期望输出 / 评分标准"，正是用前面各维度的评测指标标注出来的；指标先行，数据才有依据。这也对应《AI Engineering》"设计评估流程"一节的 Step 3——它的标题直接就是"定义评估方法**与数据**"。

**【是什么】** 把"用哪些指标、各用什么方法打分、用哪批数据喂、分数怎么映射到业务、达到什么阈值才能上线"一次性组装成一条**可自动运行、可重复回归、可卡发布**的评估流水线。

**【为什么测】** 零散地测几个指标、临时凑几条样本都没用。关键是要有一套**和业务目标对齐、数据与指标配套、能持续自动运行、能拦住高风险版本**的评估系统——此后每次改动都让它跑一遍，用数字说话。否则离线分数再高，也可能是太慢、太贵、太绕，或者安全风险没过线。

**【怎么测 / 怎么建】**（评测集与 Pipeline 一体组装）

> 定性：**一条流程、分 5 步**，严格按顺序（指标先定 → 据此标数据 → 绑业务 → 设发布闸门 → 回灌线上问题）。其中第 2 步内部还要区分两种**粒度档位**。

1. **【步骤】先锚定"测哪些组件、各用什么指标"**：把 §4 / §5 定下来的测法收拢成一张表（correctness 用功能正确性、factual consistency 用 LLM-judge、cost/latency 直接量、轨迹用按序匹配、安全用红队和护栏触发率……），并明确分层评估的粒度：per task（整个任务）/ per turn（单轮）/ per intermediate output（中间输出）。**指标先定，才知道样本要标什么。**

2. **【步骤】据此构建配套评测集**：为每条代表性输入（覆盖常见、边界、易错、"不该触发"的反例）按上一步的指标标注期望输出或评分 rubric，用 JSONL 等格式存成可批量喂入的数据集。区分两种**粒度档位**：

   * **【档位·细】test file（测试文件）**：偏单元测试，单个 case 验证某个具体行为，开发调试期高频跑；

   * **【档位·全】evalset file（评估集）**：偏集成测试，成套、覆盖多场景，用于回归。

3. **【步骤】把评估指标绑定到业务北极星指标**：比如"回答有用性分"要能关联到"用户留存 / 转化"，否则指标再漂亮也没意义。

4. **【步骤】设定上线前发布闸门**：明确多少分以上、多少成本以内、多少延迟以内、多少安全风险以内才算"可以上线"。

5. **【步骤】把线上问题回灌评测集**：生产中发现的新失败、新攻击、新成本异常，要变成新的离线 case，进入下一轮回归。

##### 5.1 发布闸门卡什么

发布闸门不是一个"总分超过 80 就上线"的粗糙判断，而是一组并列门槛。任何一类踩红线，都不应该进入真实流量。具体分为五类闸门：

* **质量 / 成功率**：上线前用 evalset、Pass@k / Pass^k、rubric、功能正确性来测，典型阈值形态是端到端成功率、关键任务 Pass^k、事实一致性分。

* **轨迹 / 工具 / 规划**：上线前用轨迹比对、工具调用静态/执行检查、规划失败模式统计来测，典型阈值形态是错工具率、错参数率、反思错误率、最大步数。

* **成本 / 效率 / 延迟**：上线前用离线批跑、压测、token 统计、工具耗时统计来测，典型阈值形态是单任务 token 预算、P95/P99 延迟、TTFT/TPOT、平均步数。

* **安全 / 合规**：上线前用越狱、直接/间接提示注入、PII/敏感数据、输出毒性、误拒率来测，典型阈值形态是红队通过率、guardrail 漏拦率、false refusal rate、审计字段完整性。

* **业务可用性**：上线前用私有业务集、用户模拟器、灰度前试运行来测，典型阈值形态是有用性阈值、投诉风险、人工兜底触发率。

这里尤其要注意：**成本 / 效率 / 延迟、安全 / 合规都属于上线前闸门**。上线后当然还要继续监控，但那不是第一次测试，而是持续验证。

##### 5.2 成本、效率、延迟

**【是什么】** 评估 Agent 完成任务"划不划算、快不快"——花了多少 token、多少钱、多少步、多长时间。

**【为什么测】** 一个准确率很高但要调用 50 次工具、烧掉几块钱、让用户等 30 秒的 Agent，在上线前就应该被拦住。质量必须和成本、延迟一起看。

**【怎么测】**

> 定性：不是步骤，而是**上线前发布闸门要量的几类指标维度**。这些指标上线后继续进入 §7.2 做真实流量监控。

* **【维度·效率】**：完成单个任务的平均步数、平均成本、单步耗时，并和基线（另一个 Agent 或人类）对比。

* **【维度·延迟】**：TTFT（首 token 时延）、TPOT（每 token 时延）、总响应时间，建议看 P95/P99 而不是只看平均值。

* **【维度·成本】**：输入 token、输出 token、工具调用次数、外部 API 费用；实战书的 `LLMInteractionMonitor` 可跟踪 token 用量来控成本。

* **【维度·发散约束】**：最大步数、重复动作、绕远路；龙猫直接把"步数过多"算作失败的一种，说明效率可以是正确性的一部分。

##### 5.3 安全与合规

**【是什么】** 评估 Agent 会不会被攻击、会不会输出有害内容、是否符合合规要求。

**【为什么测】** Agent 能调工具、能执行动作，一旦被攻破或失控，危害远大于普通聊天机器人。安全不是上线后观察一下，而是上线前红线。

**【怎么测】**

> 定性：**混合结构**——前两点是主动测试方法，第三点是要量的安全维度，第四点是治理实现。上线后这些指标继续进入 §7.2 监控。

* **▶ 方法一 · 攻击面测试**：越狱（jailbreaking）、提示注入（prompt injection，含直接/间接）、red teaming（红队主动攻击找漏洞）。

* **▶ 方法二 · 防御分层**：在 model / prompt / system 三个层面分别设防并验证；输入护栏防 PII 泄露和恶意提示，输出护栏抓无效 JSON、事实不一致、毒性、泄密等失败。

* **【维度】安全指标**：事实一致性、安全性、毒性（toxicity）、guardrail 漏拦率、guardrail 触发率、误拒率（false refusal rate）。

* **【实现】合规治理**：设计模式书提出从"Agent"升级到"高级承包商（Contractor）"——用正式"合约"明确可验证的交付物、范围、成本，把评估上升为可审计、可问责的治理机制。



***

#### 6 上线后监控

§6 已经把成本、延迟、效率、安全、合规纳入上线前发布闸门。Agent 一旦上线，评估并不结束，而是换了一种形态——**持续监控**。《AI Engineering》一句话点破：**"监控的目标和评估的目标是同一个。"** 区别只在于：上线前主要用准备好的数据、仿真和压测；上线后使用真实流量、用户反馈和系统遥测。

##### 6.1 A/B 测试与灰度发布

**【是什么】** 把新版本 Agent 只放给一部分真实用户，对比新旧版本的线上表现，再决定是否全量。

**【为什么测】** 离线评测分数高，不代表真实用户买账。A/B 测试是用真实流量做的最终裁决。注意它和 1.2 的比较式评估不同：**A/B 是把不同版本分给不同用户群看线上业务指标**，比较式评估是同一批输出给评委比。

**【怎么测】**

> 定性：**一个方法、分 3 步**（切流量 → 比线上指标 → 设回滚线），按顺序执行。

* **【步骤】** 按用户/流量切分实验组与对照组，灰度放量（先 1% 再 10% 再全量）；

* **【步骤】** 对比线上业务指标（留存、转化、满意度）而非离线分数；

* **【步骤】** 设好回滚阈值，一旦新版指标掉了立即回退。

##### 6.2 监控与可观测性

**【是什么】** 建立一套线上可观测性系统，对运行中的 Agent 做持续跟踪：它现在表现如何、有没有退化、有没有漂移、慢不慢、贵不贵、是否被攻击、能不能回溯每一步。

**【为什么测】** 上线前评测只能覆盖已知场景。真实流量会带来开发期没见过的输入、模型或数据分布变化、更长上下文、工具异常、成本放大和安全攻击。不持续盯着，等用户投诉时通常已经晚了。

**【怎么测】**

> 定性：这一节不是和"漂移 / 成本 / 安全"并列，而是它们的**统一观测底座**。先搭 logs / traces / metrics，再在这套数据上监控质量、漂移、异常、成本、延迟和安全。

1. **【底座】可观测性三件套**

   * **Logs（日志）**：记录每次请求、输入摘要、输出状态、错误、guardrail 命中、拒答原因；

   * **Traces（调用链追踪）**：记录一次 Agent 任务经过了哪些 Agent / 工具 / 模型调用，每一步耗时、token、成本和结果；

   * **Metrics（指标）**：把关键数值按时间聚合，形成 dashboard 和告警。

2. **【监控对象一】质量与行为**

   * 格式错误率：JSON 解析失败、schema 不合规；

   * 事实一致性、生成质量：可用 AI judge 抽样评估；

   * 拒答率、误拒率、guardrail 触发率；

   * 工具调用成功率、错工具率、异常工具路径。

3. **【监控对象二】漂移与异常**

   * **输入分布漂移**：用户问法、主题、query embedding 分布是否偏离历史基线；

   * **输出质量漂移**：事实一致性、满意度、拒答率是否持续变差；

   * **行为异常**：工具调用次数突然翻倍、某类任务失败率突增、Agent 出现循环；

   * **处置闭环**：发现漂移后触发重新评估 / 重新训练 / 回滚，并把线上样本回灌 §6 的评测集。

4. **【监控对象三】成本 / 效率 / 延迟**

   * **延迟**：TTFT、TPOT、总延迟、P95/P99，按用户、场景、模型版本拆分；

   * **成本**：输入/输出 token、每任务成本、每用户成本、工具/API 调用费用、缓存命中率；

   * **效率**：平均步数、最大步数、重复工具调用、工具循环、单动作耗时；

   * **处置**：超过预算时触发降级、模型路由切换、上下文裁剪、缓存策略、回滚或人工兜底。

5. **【监控对象四】安全 / 合规**

   * **攻击监控**：越狱、直接/间接提示注入、异常查询、工具越权、远程代码执行风险；

   * **护栏监控**：guardrail 触发率、漏拦率、拒答率、误拒率、人工升级率；

   * **内容与数据风险**：毒性、有害内容、事实不一致、PII / 敏感信息输入输出、品牌与合规风险；

   * **治理审计**：Contractor / 合约式治理，把可交付物、范围、成本、审计记录固定下来，保证线上行为可追责。

6. **【响应指标与工具】**

   * **MTTD / MTTR / CFR**：平均检测时长、平均修复时长、变更失败率，用来衡量团队发现和修复问题的速度；

   * **落地工具**：指标落到时序数据库（InfluxDB / Prometheus），接 Datadog / Grafana 等可观测性平台；Agent 专用的有 AgentOps，适合看交互 trace、token 和成本。

> ⚠️ **公共短板提示**：安全/合规是四份资料里整体覆盖最薄弱的一环（龙猫几乎不涉及）。真实项目里这块往往需要额外引入专门的安全测试体系，不能只靠通用评测资料。
>



**第 3 章小结**：到这里，我们已经把生命周期主线上的每一环都讲完了——从通用武器（LLM-judge、比较评估、基础度量），到上线前验证（选型 → 成功标准 → 组件 → 集成 → 评测集 / Pipeline / 发布闸门），再到上线后监控（A/B 灰度发布 → 监控与可观测性覆盖质量、漂移、成本、延迟和安全）。**下一章用美团龙猫这个真实项目，看这些思想是怎么被打包成一套完整评估系统落地的。**

***

### 4 实战：美团 LongCat VitaBench

**【背景】** 现有的 Agent benchmark 大多是"玩具任务"，和真实生活差距很大，区分不出顶尖模型到底差在哪。VitaBench 的目标是：**用贴近真实生活的复杂场景，量化现有 Agent 与实际应用需求之间的差距。**

**【场景】** 聚焦美团最熟悉的三大生活服务场景：**外卖点餐、餐厅就餐、旅游出行**。

**【为什么难】** 这些场景天然具备真实世界的复杂度：一个任务可能涉及 5–20 个服务商、100 多个候选商品，用户还会中途改主意。VitaBench 号称是"迄今最复杂的生活服务模拟环境"。

#### 4.1 整体思想

这是本章的核心。下面把**VitaBench 的每个设计，对应回第 3 章的哪个评测思想**一一列出——你会看到它几乎是第 3 章的一次"综合落地演练"：

* **4.4 规划/推理评测** → VitaBench 用**推理复杂度**维度落地：观测空间大小、部分可观测程度、推理点数量。

* **4.3 工具调用评测** → VitaBench 用**工具复杂度**落地：66 个真实工具，跨场景含 512 条依赖边的工具图。

* **5.3 多轮交互评测** → VitaBench 用**交互复杂度**加 **GPT-4.1 用户模拟器**落地，模拟用户改主意、模糊表达。

* **1.1 LLM-as-a-judge** → VitaBench 用 **Rubric 滑动窗口评估器**落地：用 Claude-7-Sonnet 当裁判，κ=0.828。

* **5.1 任务结果评测** → VitaBench 用 **Avg@4 / Pass@4 / Pass^4** 落地（每任务跑 4 次，温度 0）。

**几个特别值得讲的设计亮点：**

1. **三维复杂度框架**：VitaBench 把 Agent–用户–工具的交互建模成 POMDP（部分可观测马尔可夫决策过程），拆成**推理 / 工具 / 交互**三个可独立调节的复杂度维度。这是它最大的学术贡献——**第一次让任务复杂度变得"可控、可量化"**。

![](https://cdn.jsdelivr.net/gh/LVicBlack/IMG/root/2026/7/%E5%A4%A7%E6%A8%A1%E5%9E%8B%E5%BA%94%E7%94%A8!%E7%AE%97%E6%B3%95%E5%AD%A6%E4%B9%A0%E8%B7%AF%E7%BA%BF+%E5%85%AB%E8%82%A1+%E9%9D%A2%E8%AF%95%E5%AE%9E%E6%88%98_3-image-17.png)

* **Rubric 滑动窗口评估器**：把任务目标拆成一个个原子化的判定准则，用"全有或全无"方式严格判定，和人类评判的一致性高达 Cohen's κ=0.828——说明这套自动评估很可信。这正是 1.1 LLM-as-a-judge 的高质量工程实现。

* **故意去掉"规则文档"**：传统 benchmark 会给 Agent 一份 policy 文档说明规则，VitaBench 反其道而行——**把领域规则编码进工具依赖图里，逼模型自己推理**，而不是照着说明书抄。这让评测更接近真实（现实中没人给你完整说明书）。

* **真实工具图**：66 个真实工具之间有 512 条依赖边（调用 A 工具前必须先调 B），构成一张复杂的依赖网络。

#### 4.2 结果与启示

**【结果很扎心】** 即便是当时最强的模型 o3，在 VitaBench 上：

* 跨场景 **Avg@4 仅 30%**，单场景 < 50%；

* **Pass^4 ≈ 0%**（连续 4 次全做对的概率几乎为零，说明极不稳定）；

* 失败主因是**推理错误（占 61.8%）**。

**【启示】**

1. **顶尖 Agent 离"真实可用"还很远**——在玩具 benchmark 上的高分有水分，一到真实复杂场景就现原形。

2. **推理是当前最大瓶颈**（61.8% 失败源于推理），印证了 4.4 规划/推理评测的重要性。

3. **稳定性比单次成功更难**——Pass^4≈0% 说明 Agent 偶尔做对容易、稳定做对极难，这也是为什么龙猫要用 Pass^k 这种严格指标。
